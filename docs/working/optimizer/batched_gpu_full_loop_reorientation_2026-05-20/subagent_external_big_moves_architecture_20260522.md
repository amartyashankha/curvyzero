# External Big-Move Architecture Critique

Date: 2026-05-22

Scope: fresh external-architecture critique for the CurvyTron optimizer
reorientation. This note focuses on architecture-level moves, not tactical
wrappers. No live runs, trainer defaults, tournament defaults, or code were
changed.

Local premise taken seriously:

```text
direct_ctree_gpu_latent is only about 1.3x in matched full-loop rows because
the LightZero collect/search/dataflow topology still crosses Python, CPU,
NumPy/list, CPU-CTree, and scalar packaging boundaries.
```

## Core Judgment

The big external systems do not get their speed by polishing a scalar collect
call. They change the unit of work:

```text
many active games/roots
-> batched leaf/root inference
-> compiled or service-owned tree/search state
-> compact trajectory/replay records
-> learner fed asynchronously or at a coarse boundary
```

For CurvyTron stock fixed-opponent first, the practical translation is:

```text
many fixed-opponent env rows
-> batch only the learner-seat search roots first
-> return compact action / visit / value arrays
-> let the existing fixed opponent and replay contract remain the edge
```

Do not make the first big move a full self-play league, a new tournament
system, or a wholesale learner rewrite. The first big move should own the
search boundary for fixed-opponent collection.

## Source Signals

- AlphaZero used self-play games generated by MCTS and trained a network from
  those search policies. The reported run used 5,000 first-generation TPUs for
  self-play generation and 64 second-generation TPUs for training:
  <https://arxiv.org/pdf/1712.01815>.
- MuZero generated data with the latest checkpoint, used 800 simulations per
  move for board games and 50 for Atari, and kept replay buffers of recent
  games/sequences. The paper reports 16 TPUs for board-game training and 1,000
  TPUs for board-game self-play:
  <https://arxiv.org/pdf/1911.08265>.
- OpenSpiel's AlphaZero docs split actors, neural evaluator, learner, and
  evaluators; its faster C++ path uses threads, a shared cache, batched
  inference, and GPU support:
  <https://openspiel.readthedocs.io/en/latest/alpha_zero.html>.
- MiniZero has server, self-play workers, optimization worker, and storage.
  Each self-play worker keeps multiple MCTS instances and batches leaf
  evaluation on GPU:
  <https://github.com/rlglab/minizero>.
- KataGo's analysis engine gets speed from cross-position batching and an NN
  cache:
  <https://github.com/lightvector/KataGo/blob/master/docs/Analysis_Engine.md>.
  Its self-play acceleration paper reports a 50x compute reduction versus
  comparable AlphaZero-like methods through algorithm and architecture changes:
  <https://arxiv.org/abs/1902.10565>.
- MCTX is JAX-native, JIT-compatible, and batch-first:
  <https://github.com/google-deepmind/mctx>.
- EfficientZero improves MuZero sample efficiency and its public repo exposes
  parallel env count, GPU actor count, replay priority, and reanalyze policy
  controls:
  <https://github.com/YeWR/EfficientZero>,
  <https://arxiv.org/abs/2111.00210>.
- SpeedyZero is especially relevant as systems evidence: it partitions
  EfficientZero-style work into data collection, batch reanalysis, and training
  nodes, with a reported 14.5x acceleration over EfficientZero:
  <https://sites.google.com/view/speedyzero>.
- LightZero already uses C++ CTree internals; its docs say MuZero CTree core
  `batch_traverse` and `batch_backpropagate` are C++, and EfficientZero CTree
  is also C++ batch format:
  <https://opendilab.github.io/LightZero/api_doc/mcts/tree_search/index.html>.

## Practical Ranking For CurvyTron Fixed-Opponent

### 1. MiniZero/KataGo-Style Local Search Service

This is the best "big move" to design first.

The first implementation should not be a distributed self-play system. It
should be a single-node service-shaped collector:

```text
fixed-opponent env rows
-> learner-seat observations and legal masks
-> search service owns roots/trees and recurrent batching
-> compact action, visit, root value tensors
-> existing replay/LightZero-compatible rows at the edge
```

Why first:

- It attacks the topology that is actually limiting CurvyTron.
- It matches MiniZero and KataGo more closely than another direct CTree hook.
- It keeps stock fixed-opponent semantics as the first acceptance target.
- It lets us measure queue fill, batch latency, GPU occupancy, and replay
  materialization separately.

The smallest useful falsifier is a mock service ceiling:

```text
same env rows and legal masks
-> no LightZero CTree
-> compact synthetic or prerecorded visits/values/actions
-> existing replay edge
```

If that ceiling is not large, a real service will not be a `5x-10x` answer.

### 2. Fixed-Shape Compiled Dense Search As The Search-Core Prototype

This is still worth doing, but frame it as the search core for a future service,
not as another wrapper.

Keep it fixed:

```text
N = active learner roots
A = 3
sim = fixed
root_noise = 0 for first speed gate
all buffers preallocated
masks instead of shape changes
```

It validates whether CurvyTron's tiny action space can use a device-resident
tree effectively. Kill it quickly if sim16 does not beat the current
`direct_ctree_gpu_latent` denominator.

### 3. Array-Native CTree Only As A Compatibility Bridge

Array-native CTree is useful if the project needs a lower-risk bridge while the
service is being designed. It should not be sold as the big architecture. It
still leaves search CPU-owned and recurrent inference synchronized through
Python/PyTorch each simulation.

### 4. MCTX/JAX As A Scratch Truth Serum

MCTX is architecturally clean. It is also not a patch. A meaningful MCTX lane
means JAX model/search ownership, not a PyTorch model called from JAX. Use it if
the question becomes "what is the all-device ceiling?", not "what can we ship
inside the current LightZero coach?"

### 5. EfficientZero/SpeedyZero Later, Not First

EfficientZero and SpeedyZero are relevant because they show that MuZero-family
speed comes from pipeline ownership and data/reanalysis architecture, not only
from faster search kernels. But CurvyTron's immediate wall is still the
fixed-opponent collect/search boundary. Do not add reanalysis or multi-node
trainer machinery before the collector can emit compact data quickly.

## Recommended Next Big-Move Plan

1. **Write the search-service contract first.**
   Define the minimal fixed-opponent API:

   ```text
   input:  obs_uint8[N,4,64,64], legal_mask[N,3], root_meta[N]
   output: action[N], visits[N,3], root_value[N], policy_debug[N,3]
   edge:   existing replay row materialization
   ```

2. **Run a mock service ceiling.**
   It should bypass CTree and return compact legal actions/visits/values. This
   tests whether removing LightZero-shaped search/output packaging can actually
   expose a `5x-10x` full-loop ceiling.

3. **Use compiled dense search or array-native CTree as interchangeable search
   cores behind the same service contract.**
   The service boundary matters more than which core wins first.

4. **Only then consider actor farms.**
   Add multiple fixed-opponent collectors feeding the service once one collector
   proves the service can batch and replay compactly.

5. **Keep stock LightZero training as the oracle until parity gates pass.**
   The first target is stock fixed-opponent output equivalence and training
   stability, not self-play league generality.

## Hard Invalidation Tests

Kill or downgrade the radical architecture thesis if any of these are true:

- A mock search-service ceiling is not materially above current
  `direct_ctree_gpu_latent` full-loop throughput.
- With CTree removed from the hot path, env stepping, opponent logic, replay,
  RND, or learner becomes the dominant wall before reaching the target
  multiplier.
- Fixed-shape dense sim16 cannot beat the current direct GPU-latent row and
  profiling shows the loss is inherent recurrent/model work, not launch
  overhead.
- Multi-root batching cannot fill the GPU without unacceptable queue latency.
- Compact service outputs cannot reproduce stock fixed-opponent replay fields,
  legal-action behavior, root values, and visit targets.

## Bottom Line

The big move is not "more CTree glue" and not "rewrite everything in JAX".

The big move is:

```text
own the fixed-opponent search service boundary
```

Start with a mock service ceiling and a fixed-shape search-core prototype behind
the same compact API. If the ceiling is high, the external systems say the
MiniZero/KataGo pattern is the right production direction. If the ceiling is
low, the bottleneck has already moved elsewhere and a radical search rewrite
would be theater in a very convincing lab coat.
