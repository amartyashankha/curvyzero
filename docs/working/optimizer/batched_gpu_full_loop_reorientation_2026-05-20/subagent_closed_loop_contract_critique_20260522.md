# Closed Compact Loop Contract Critique

Date: 2026-05-22

Status: narrow optimizer critique. Read-only with respect to live Coach runs.

## Plain Answer

The new `--closed-loop-steps` metric is a valid next blocker for the optimizer.
It is the first MCTX/JAX probe that repeatedly closes this loop:

```text
real compact visual roots
-> MCTX search
-> selected actions
-> compact env step
-> CompactReplayIndexRowsV1
-> repeat
```

That is the right denominator after the earlier fresh-boundary MCTX numbers,
because the earlier numbers did not pay the next env/observation/replay edge.

But it is still not a trainer-speed claim. It is a profile-only blocker probe.

## What Can Still Mislead Us

1. The search model is a toy JAX MuZero model, not the actual LightZero PyTorch
   model, weights, heads, support transforms, or temperature/noise schedule.
   The metric can prove the compact loop shape is fast or slow; it cannot prove
   the production trainer will learn or match stock search semantics.

2. The replay edge is index-only. That is the intended hot-path shape, but it
   does not include replay storage, sampling, target unroll materialization,
   learner tensor assembly, learner updates, or RND bonus/update cost.

3. The loop excludes the initial compact-manager setup and first rollout
   construction from the closed-loop denominator. That is fine for steady-state
   search/service work, but misleading if reset/autoreset or manager setup is
   frequent in real runs.

4. `active_roots_per_sec` can hide inactive-row overhead. If many rows die, the
   benchmark may still spend time carrying fixed-size arrays while the numerator
   shrinks. We need active roots, total roots, env rows, terminal/autoreset
   counts, and completed-step counts reported together.

5. Inactive roots get a default joint action before the env step. The current
   code overwrites active seats with search actions, but we still need a hostile
   inactive-root test proving inactive/default actions are ignored or harmless.

6. Terminal/final reward semantics are not fully stressed. The benchmark passes
   `next_final_reward_map=next_step.reward`, so it does not catch a case where
   terminal final reward differs from immediate reward.

7. The loop validates `CompactReplayIndexRowsV1`, but it does not materialize
   those rows against a real multi-record replay chunk during the profile. The
   local tests cover materialization separately; the benchmark still only proves
   the collection-edge shape.

8. The timing includes explicit H2D and D2H action/policy readback, which is
   good, but it also serializes search, readback, env step, and replay writing.
   That may understate a future async service. It may also overstate production
   speed if hidden scalar reads appear later. The residual between per-step
   bucket sums and total wall should be tracked.

9. The renderer/backend contract must fail closed. This metric is meaningful
   only when it is using the real compact visual surface we intend to optimize,
   not CPU oracle, fake roots, stale resident roots, or old body-circle aliases.

10. The action distribution is generated by a toy policy. It may not stress
    deaths, bonuses, no-op/dropout stochasticity, long trails, or weird masks
    the way a real current policy does.

## Checks To Add Next

P0 checks before using this as a serious optimizer gate:

- Add a tiny local test that exercises `closed_loop_steps` enough to prove the
  result reports `completed_steps`, `total_active_roots`, per-step timing
  buckets, and a non-null `active_roots_per_sec`.
- Add an adversarial inactive-root test: non-prefix active roots plus inactive
  seats, then prove default inactive actions do not affect replay rows.
- Add a terminal/autoreset test where `final_observation_row_mask=true` and
  `next_final_reward_map != next_reward`, then prove index rows keep the final
  reward flag and materialize correctly.
- Add a hostile timing guard: per-step
  `root_build + h2d + search + d2h + env_step + replay_index` should be close
  to `total_sec`, with residual reported.
- Add a metadata fail-closed check for renderer backend, render surface,
  bonus-symbol mode, observation shape/dtype, perspective, `to_play`, reward
  schema, and `num_simulations`.
- Add a compact RND edge check: latest-frame extraction from compact
  observations plus a measured no-RND vs `rnd_meter_v0` bucket.
- Add stress rows with enough warmup and repeated closed-loop steps:
  B256/B512/B1024, sim16/sim32, normal death and no-death, H100 plus at least
  one L4 sanity row.

P1 checks before trainer-facing recommendations:

- Use the actual current model path or an export that makes the JAX/MCTX model
  cost comparable to the PyTorch LightZero model.
- Compare selected actions and visit targets against stock/direct CTree on
  forced-mask and clear-preference cases, not neutral tie-heavy exact parity.
- Materialize a compact replay sample into learner-shaped target rows over a
  multi-record chunk and compare to the existing target-row builder.
- Run a matched stock/full-loop profile after the closed-loop probe identifies
  which bucket is hot.

## Recommendation

Use the closed compact loop as the next optimizer blocker. It is the right
place to learn whether MCTX/device-resident search is actually useful once we
pay env, observation, and replay-index costs.

Do not quote its roots/sec as full training speed. Quote it as:

```text
profile-only closed compact search/env/replay-index throughput
```

The decision rule should be simple:

- If closed-loop wall is mostly search, keep pushing MCTX/search-service work.
- If closed-loop wall moves to env/observation/replay/RND, stop polishing
  search and attack that bucket.
- If the closed loop lands near current direct CTree full-loop speed, the next
  move is a larger architecture change, not another small wrapper patch.
