# Optimizer Blockers

Date: 2026-05-09

Status: working blocker list. This is not an environment-fidelity or
training-quality report.

## Blocking A Production Speed Claim

- No full CurvyTron actor loop yet: real reset/autoreset, trainer-facing
  observations/rewards, final observations, replay writing, learner handoff,
  and policy/search timing have not run together.
- The current loop is not production-shaped: reset/lifecycle/autoreset helpers
  exist, but they are not yet one continuous source-faithful training runtime.
- Fast path parity is still contract-scoped. It is tied to promoted source
  contracts and fixture slices, not broad natural multiplayer rollout.
- Current actor-loop benchmarks still use debug observations, synthetic
  feedback, synthetic NumPy policy/search, and mostly in-memory replay staging.
- The actor-loop bridge can write replay-v0-shaped sample chunks, but those
  chunks are explicitly blocked for training because they carry debug
  observation/reward schemas rather than trainer-facing rays and sparse payoff
  rewards.
- Production replay cost is unknown. Array staging looks cheap; `.npz`/Volume/
  object-store/learner-adjacent writes are not yet profiled.
- Real policy/search cost is unknown. Synthetic stand-ins and Modal/Mctx smokes
  do not answer PPO forward cost, LightZero MCTS cost, or project-owned Mctx
  search inside the real loop.
- Completed games/minute and p95/p99 action latency are not yet measured on a
  comparison-valid CurvyTron loop.

## P0 Unblockers

- Source-stepped trainer-facing no-train harness now exists and has passed
  locally with `observation[T,B,P,106]`, sparse rewards, row mapping, and
  replay-v0 write/read. The current best local mode uses source body-circle rays
  and avatar body sidecar metadata. Remaining blocker:
  browser-visible trail/bonus/final observation/public-autoreset semantics plus
  real policy/search/learner timing.
- Replace synthetic policy/search timing with at least one real policy forward
  timing on the same ego-row shape.
- Record replay write timing and bytes/chunk instead of only in-memory staging.
- Standardize debug mode: full debug events for fidelity/probes, sampled or
  disabled debug events for hot-loop profiles.

## Blocking Setup Choice

- No project-owned PPO/CleanRL-style CurvyTron runner exists yet.
- No project-owned MuZero/Mctx trainer exists yet; Mctx is still a search/runtime
  boundary spike.
- CurvyTron is not yet a real registered LightZero training env. Current
  LightZero evidence is stock controls, dummy Pong bridge, and adapter smoke.
- Training lane evidence is mixed: official Atari mechanics work but signal is
  weak/blocked by setup quality; custom dummy Pong still has action-collapse and
  target-quality blockers.

## Blocking Systems Rewrites

- CPU env stepping is not proven to be the final bottleneck once real model/
  search and replay write costs are included.
- GPU env, JAX env, C++/Rust, EnvPool-style backends, distributed actors, and
  central inference are premature until the actor-loop profile points there.
- Debug event rows are a large hot-path tax, but the production debug/sampling
  policy is not decided.
- Batch size policy is open: larger batches may improve throughput while hurting
  p95/p99 action latency or policy freshness.

## Unknowns To Mark Explicitly

- First comparison-valid profile shape.
- Real PPO model forward/update cost on trainer-facing observations.
- Real MCTS/Mctx cost, transfer cost, and memory shape on ego rows.
- Replay write path and chunk size.
- Policy-staleness budget for actor pools or async collection.
- Whether LightZero, owned PPO, or owned Mctx becomes the first serious runner.

## Not Blockers By Themselves

- Stock Atari/CartPole controls are useful controls, not CurvyTron proof.
- Synthetic Mctx/Modal smokes prove dependency/runtime boundaries, not rollout
  throughput.
- High fixture rows/sec is useful shape evidence, not training speed.
- Source-env scout speed is useful narrow evidence, not final fast-path proof.
