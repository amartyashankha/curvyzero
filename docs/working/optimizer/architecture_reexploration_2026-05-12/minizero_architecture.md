## Short Verdict

MiniZero is a serious full-system Zero training stack, not a toy learner loop.
It has the architecture shape CurvyTron probably needs: many self-play games,
batched neural inference inside actors, file-backed replay/model storage, a
central coordinator, and a separate optimizer worker.

It is not a drop-in base for visual simultaneous CurvyTron. The core game
contract is one action by the current player, then the environment advances.
The Atari path proves visual MuZero machinery, but it is single-player and
Atari-shaped. A CurvyTron port is plausible, but the integration cost is high
unless we accept a centralized joint-action control first.

Current practical read: use MiniZero as an architecture/control reference before
treating it as a migration target.

## Process Model

MiniZero training has four roles:

- `zero_server`: C++ TCP coordinator.
- `sp` workers: C++ self-play workers.
- `op` worker: Python/PyTorch optimizer.
- storage: training directory with `sgf/`, `model/`, logs, and config; NFS is
  the documented distributed storage option.

The server loop is synchronous by iteration:

1. Tell connected self-play workers to load the latest `weight_iter_*.pt`,
   reset actors, and start.
2. Collect `zero_num_games_per_iteration` self-play records from workers.
3. Stop self-play workers.
4. Tell the optimizer worker to train from the SGF window
   `[iteration - zero_replay_buffer + 1, iteration]`.
5. Wait for `Optimization_Done`.
6. Publish the next model in `model/` and repeat.

Workers are simple processes connected to the server over TCP using
`scripts/zero-worker.sh`. The recommended pattern is one self-play worker per
GPU and at most one optimizer worker using all GPUs. `tools/quick-run.sh` starts
server, self-play workers, and optimizer for local runs.

Important implication: MiniZero's distributed design is coarse and practical.
It does not require Ray/Reverb/etc. The coordination layer is small, but the
self-play worker itself is deeply tied to the C++ environment/search/network
interfaces.

## Self-Play And GPU Batching

Each self-play worker runs an `ActorGroup`.

Inside one worker:

- `zero_num_parallel_games` actor instances run at once.
- Each actor owns an environment and MCTS tree.
- One TorchScript model is loaded per visible GPU, up to the number of parallel
  games.
- Actors are assigned to models by actor index modulo model count.
- CPU phases do MCTS selection, env stepping, target bookkeeping, and push
  inference requests.
- GPU phases run one batched forward pass per model/GPU for the queued requests.

For MuZero, root nodes use `initial_inference`; non-root nodes use
`recurrent_inference`. The worker batches across games, not across machines.
This is a useful pattern to copy: many active games per GPU, gather leaf
requests, run one model batch, then resume tree work.

The relevant knobs are:

- `zero_num_parallel_games`: active games per self-play worker.
- `zero_num_threads`: CPU worker threads in the self-play process.
- `actor_num_simulation`: MCTS simulations per move.
- `actor_mcts_value_rescale`: value normalization for larger reward scales.
- `actor_select_action_by_count`,
  `actor_select_action_by_softmax_count`,
  `actor_select_action_softmax_temperature`,
  `actor_select_action_softmax_temperature_decay`.
- `actor_use_dirichlet_noise`,
  `actor_dirichlet_noise_alpha`,
  `actor_dirichlet_noise_epsilon`.
- `actor_use_gumbel`,
  `actor_use_gumbel_noise`,
  `actor_gumbel_sample_size`,
  `actor_gumbel_sigma_visit_c`,
  `actor_gumbel_sigma_scale_c`.

The MCTS tree allocation is roughly
`(actor_num_simulation + 1) * action_size` nodes per actor, so action-space
growth matters directly.

## Optimizer And Replay

The optimizer is Python/PyTorch. It reads SGF records from disk, parses them
through a C++ pybind data loader, maintains an in-memory replay buffer, samples
batches, trains, then saves both `.pkl` and TorchScript `.pt` weights.

Important knobs:

- `zero_num_games_per_iteration`: data generated before each optimizer phase.
- `zero_replay_buffer`: number of iterations kept in replay.
- `learner_training_step`: optimizer steps per iteration.
- `learner_batch_size`: training batch size.
- `learner_muzero_unrolling_step`: MuZero unroll length.
- `learner_n_step_return`: n-step bootstrap horizon.
- `learner_use_per` and related PER beta/alpha controls.
- `learner_optimizer`, `learner_learning_rate`, `learner_weight_decay`.

The server can accept self-play generated by older models via
`zero_server_accept_different_model_games`. Turning that off is a clean
freshness control.

## Atari And Visual Support

MiniZero's visual proof is Atari:

- ALE environment in C++.
- RGB screen resized with OpenCV to `96x96`.
- Feature stack of 8 history steps.
- Each history step contains one action plane plus 3 RGB planes, so the network
  input is `32 x 96 x 96`.
- Action size is fixed at 18.
- MuZero action features are one-hot action planes over hidden spatial size.
- Hidden spatial size is `6 x 6` for the Atari network.
- Value and reward are categorical with support size 601.
- Atari defaults set `learner_n_step_return=10` and
  `zero_actor_intermediate_sequence_length=200`.
- Observations are compressed into SGF records; long visual trajectories can be
  emitted as chunks instead of full episodes.

This is real visual MuZero support. It is not generic visual multi-agent
support. The Python network factory chooses the Atari MuZero network when the
game name contains `atari`; otherwise `nn_type_name=muzero` uses the board-like
MuZero network. A CurvyTron visual port would need either a clean CurvyTron
visual network factory branch or an intentionally Atari-shaped compatibility
path.

## Simultaneous CurvyTron Fit

MiniZero is naturally turn-based.

The base environment API exposes a current `turn`, an `Action(player,
action_id)`, `act(action)`, `getLegalActions()`, and `getNumPlayer()`. The
player enum and helpers only support one or two players. The actor performs one
MCTS, chooses one action, calls `env.act(action)`, records one policy/value/reward
entry, and repeats.

That creates three possible CurvyTron approaches:

1. Centralized joint action.
   Encode a 2-player physical tick as one action, e.g. `3 * 3 = 9`. This is the
   easiest MiniZero port and would preserve the visual/delayed-reward machinery.
   It trains a centralized controller, not independent self-play.

2. Sequential commit wrapper.
   Let player 0 choose a pending action, then player 1 choose a pending action,
   then advance the world. This matches MiniZero's turn machinery but risks
   changing the game. Pending actions must be hidden from features, rewards, and
   target records. Even then, the replay would contain half-tick policies unless
   the actor/loader are changed.

3. Independent per-seat search from one physical snapshot.
   Run two root searches from the same state, collect both policies, commit the
   joint action, then record one physical tick. This is the best semantic fit
   for near-term CurvyTron self-play, but it requires custom actor and replay
   work. MiniZero does not provide this out of the box.

For final CurvyTron semantics, option 3 is the interesting one. For a framework
control or first migration smoke, option 1 is much cheaper.

## Integration Cost

Minimum credible CurvyTron port:

- Implement a C++ CurvyTron `Action`, `Env`, and `EnvLoader`.
- Add it to `environment.h`, `environment/CMakeLists.txt`, and `scripts/build.sh`.
- Port or rewrite the CurvyTron physics and raster observation path in C++.
- Decide observation shape, history stack, action features, reward scaling, and
  value/reward target format.
- Add a network factory path for visual CurvyTron, likely Atari-style MuZero but
  with CurvyTron dimensions.
- Validate SGF/storage size for visual trajectories.
- Build/run on Linux with NVIDIA/CUDA, libtorch, PyTorch, Boost, OpenCV, and the
  MiniZero container assumptions.

Extra cost for proper simultaneous independent self-play:

- Change `BaseActor`/`ZeroActor` or create a CurvyTron-specific actor that runs
  two searches per physical tick.
- Store two root policy targets per tick or one per-seat trajectory view.
- Ensure `EnvLoader` samples targets by physical tick/seat, not artificial
  half-turn.
- Audit value perspective and reward assignment.
- Add order-invariance and leak canaries.

This is a substantial migration. The architecture is attractive, but the
semantic bridge is not small.

## Controls Worth Replicating

Useful MiniZero runs:

- Atari MuZero or Gumbel MuZero quick run, preferably `pong` if the ROM is
  available, otherwise the documented `ms_pacman` command. This tests the full
  visual server/self-play/optimizer/storage loop.
- TicTacToe/Othello/Gomoku quick run. This tests the two-player value/policy
  path without visual observations.
- Batch ladder on Atari: vary self-play `-b` / `zero_num_parallel_games` and
  `-c` / `zero_num_threads` while holding `actor_num_simulation` fixed. Measure
  self-play time, optimizer time, GPU utilization, and SGF throughput.
- Freshness control: compare
  `zero_server_accept_different_model_games=true/false`.
- Chunked visual replay control: inspect Atari SGF size and load time with
  `zero_actor_intermediate_sequence_length=200`.

Specific architecture controls to copy into CurvyTron/LightZero work:

- Many live games per GPU, not one game loop per model call.
- Explicit leaf-request batching before GPU inference.
- Separate self-play and optimizer phases with checkpoint publication.
- File-backed trajectory chunks that can be generated by many workers and later
  loaded by one learner.
- Configurable replay window and train/data ratio:
  `zero_replay_buffer`, `zero_num_games_per_iteration`,
  `learner_training_step`.
- Action-selection/noise controls: Dirichlet, Gumbel, count vs softmax-count,
  temperature decay.
- Model freshness switch for accepting or discarding stale actor output.

## Practical Recommendation

Do not migrate to MiniZero yet.

First, replicate one MiniZero visual full-loop control and one MiniZero
two-player board control. In parallel, copy the actor batching idea into our
architecture sketch: keep many searched roots alive, batch model calls, write
trajectory chunks, and let the learner consume chunks from a checkpointed
iteration boundary.

MiniZero becomes a stronger base candidate only if a small C++ visual CurvyTron
env can run under the Atari-style MuZero path and if we are willing to start
with centralized joint action. If the goal is correct independent simultaneous
self-play, MiniZero is more valuable as a reference than as a shortcut.
