# Granular Action Cadence Audit - 2026-05-13

## Verdict

The game/runtime can support one policy action per granular CurvyTron source physics step, but the trusted training lane does not currently default to that cadence.

The trusted lane is `src/curvyzero/infra/modal/lightzero_curvyzero_stacked_debug_visual_survival_train.py --mode train`. It imports stock `lzero.entry.train_muzero` and calls it directly in `_run_visual_survival_train` (`src/curvyzero/infra/modal/lightzero_curvyzero_stacked_debug_visual_survival_train.py:3454`, `src/curvyzero/infra/modal/lightzero_curvyzero_stacked_debug_visual_survival_train.py:3642`, `src/curvyzero/infra/modal/lightzero_curvyzero_stacked_debug_visual_survival_train.py:3755`).

The current source-state visual survival default is a 12-source-frame decision interval: `DEFAULT_DECISION_MS` comes from `DEFAULT_SOURCE_FRAME_DECISION_MS`, which is `SOURCE_PHYSICS_STEP_MS * DEFAULT_DECISION_SOURCE_FRAMES`, and `DEFAULT_DECISION_SOURCE_FRAMES = 12` (`src/curvyzero/env/vector_multiplayer_env.py:51`, `src/curvyzero/env/vector_multiplayer_env.py:52`, `src/curvyzero/env/vector_multiplayer_env.py:53`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:236`, `src/curvyzero/infra/modal/lightzero_curvyzero_stacked_debug_visual_survival_train.py:353`). At 60 Hz, that is about 200 ms per LightZero action.

So: unless `decision_ms` is explicitly set to one source physics frame, or cfg supplies `decision_source_frames=1`, one policy action covers multiple internal source ticks. That violates the new Coach requirement as a default.

## Repeat / Multi-Tick Knobs

- `decision_ms`: current trusted CLI knob. If larger than one source physics tick, it creates a multi-source-frame LightZero decision. The wrapper enforces whole-number source frames but does not require that number to be 1 (`src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:2707`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:2711`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:2713`).
- `decision_source_frames`: accepted by source-state env cfg but not exposed/threaded by the trusted launcher. Setting it to `1` is the cleanest granular mode (`src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:2705`, `src/curvyzero/env/vector_multiplayer_env.py:281`, `src/curvyzero/env/vector_multiplayer_env.py:292`).
- `policy_action_repeat_min/max/extra_probability`: explicitly repeats one policy action across multiple wrapper/vector steps. Defaults are currently 1/1/0, but non-default values multiply the already-bundled `decision_source_frames` cadence (`src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:250`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:594`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:1421`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:1644`).
- `source_max_steps`: despite the name, in this wrapper it is the configured LightZero decision-step cap before multiplying by `decision_source_frames`; the underlying vector `max_ticks` becomes `source_max_steps * decision_source_frames` (`src/curvyzero/infra/modal/lightzero_curvyzero_stacked_debug_visual_survival_train.py:4556`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:530`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:533`, `src/curvyzero/env/vector_multiplayer_env.py:2671`).
- Legacy `CurvyTronEnv` has `CurvyTronConfig.action_repeat=4` by default and loops inside `step`, but that is not the trusted source-state visual survival lane (`src/curvyzero/env/config.py:98`, `src/curvyzero/env/config.py:99`, `src/curvyzero/env/core.py:137`).
- The scalar source oracle supports action every arbitrary elapsed frame: `CurvyTronSourceEnv.step` applies joint actions, then advances one `elapsed_ms` frame. Passing `elapsed_ms=tick_ms` gives granular source cadence (`src/curvyzero/env/source_env.py:711`, `src/curvyzero/env/source_env.py:725`, `src/curvyzero/env/source_env.py:726`, `src/curvyzero/env/config.py:55`).

## Critical Notes

- Current telemetry names are easy to misread: wrapper `source_tick_index` is based on `_physical_step_index`, but one "physical step" may be a vector step containing 12 source physics substeps (`src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:1629`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:1630`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:1650`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:1654`).
- The joint-action variant claims/rejects based on "exactly one source tick per step", but its guard only checks policy repeat, not `decision_source_frames`; existing product-fidelity tests intentionally use 12 frames (`src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:2377`, `src/curvyzero/training/curvyzero_source_state_visual_survival_lightzero_env.py:2382`, `tests/test_lightzero_source_state_wrapper_product_fidelity.py:37`, `tests/test_lightzero_source_state_wrapper_product_fidelity.py:101`). This is outside the fixed-opponent trusted lane, but it is the same cadence hazard.
