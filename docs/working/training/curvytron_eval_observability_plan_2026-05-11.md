# CurvyTron Eval And Observability Plan - 2026-05-11

Purpose: define the simple eval suite we need before more training claims.

## North Star

For now, CurvyTron learning means this:

> Later checkpoints survive longer than earlier checkpoints under the same eval
> settings, without collapsing to one action.

Survival time is the main signal. Everything else supports that signal.

## Minimum Eval Panels

### 1. Run Summary

Show what was actually run:

- run id and attempt id
- training path
- current-policy self-play or fixed opponent
- reward settings
- `decision_ms`
- max ticks
- checkpoint cadence
- progress cadence
- compute target
- latest iteration
- latest checkpoint

### 2. Survival Curve

For each checkpoint:

- mean survival
- median survival
- max survival
- completed episode count
- capped episode count
- failure count

Use one common random seed set per eval bundle. The seed set can be generated by
a reproducible random number generator, but the point is to sample many starts,
not chase one fixed seed.

### 3. Action Mix

For each checkpoint:

- action histogram
- top action fraction
- per-player action histogram when available
- warning if one action dominates

This catches the simple failure mode where the policy always turns the same way
or always goes straight.

### 4. Death Reasons

For each checkpoint:

- terminal reason counts
- death player counts
- wall vs body vs timeout when available
- draw count

If the current artifacts do not expose wall/body owner clearly, record that as a
missing field and add it next.

### 5. Episode Traces

For a small number of episodes:

- reset summary
- step count
- actions
- final reason
- short step snippets near death
- optional frame/contact-sheet later

The goal is not beauty. The goal is to quickly answer why an episode ended.

## Simple Baselines

Use these as sanity checks:

- straight
- random legal
- wall avoid
- ray clearance

At `decision_ms=300`, simple policies can die after a small number of decisions
because each decision moves the player far. That is expected. Compare trained
checkpoints against baselines at the same timestep.

## Report Warnings

The inspector should be blunt when data is not trustworthy:

- missing progress file
- missing eval manifest
- latest checkpoint not evaluated
- mixed `decision_ms`
- mixed map size
- mixed opponent type
- fixed-opponent result labeled as self-play
- no strict checkpoint load
- action collapse
- no completed episodes
- timeout caused by too-small max ticks
- no death-cause detail

## Durable Output

Write one durable report per eval bundle:

```text
artifacts/local/curvytron-inspector/<run_id>/<eval_id>/report.json
artifacts/local/curvytron-inspector/<run_id>/<eval_id>/report.md
```

Keep the report compact, but include artifact refs so a later agent can fetch the
raw files.

## Later Additions

Only add these after the first report is useful:

- tiny images or contact sheets near death
- nearest wall/trail distances
- player path length in map units
- MCTS visit distributions
- comparison between first, middle, latest, and best checkpoints
- side-by-side comparison across training variants

## Passing Gate

The first useful inspector version passes if a human can answer these questions
in under a minute:

- Which run is this?
- Which checkpoints exist?
- Did survival improve?
- Did actions collapse?
- How did episodes usually end?
- Are the comparisons fair?
- What should be inspected next?
