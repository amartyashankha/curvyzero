# MuZero Training Pitfalls Literature

Purpose: concise source-backed checklist for the CurvyTron no-learning failure.
Read this as "what could silently make weights move while policy quality does
not improve."

## Narrow Source Scan: CurvyTron-Relevant Contracts

This scan intentionally sticks to primary or near-primary sources: the MuZero
and AlphaZero papers/pseudocode, DeepMind/OpenSpiel/MCTX implementation docs or
source, LightZero source docs, and the transformed-support paper used by MuZero
Atari-style value/reward heads.

| Local question | Source-backed contract | CurvyTron pitfall to audit |
| --- | --- | --- |
| Replay target construction | MuZero trains an unrolled model on policy, value, and reward targets at actual future timesteps; the value target is observed rewards plus a bootstrapped future search value, the reward target is the observed reward, and the policy target is produced by MCTS. See the [MuZero paper](https://ar5iv.labs.arxiv.org/html/1911.08265) and the arXiv ancillary [`pseudocode.py`](https://arxiv.org/src/1911.08265v1/anc/pseudocode.py). | A shifted replay row can still produce changing weights while each head learns the wrong timestep. Write a tiny terminal/near-terminal trajectory parity test for `(value, reward, policy)` at every unroll index. |
| `to_play` and value perspective | MuZero pseudocode carries `to_play` through search and backup; backup signs depend on whether a node is for the same player. MCTX's generic backup is only `reward + discount * leaf_value`; perspective/sign convention must come from the environment/model contract, not from MCTS magic. See MuZero [`pseudocode.py`](https://arxiv.org/src/1911.08265v1/anc/pseudocode.py) and [MCTX search backup](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/search.py#L1414-L1483). | For CurvyTron, decide ego/current-player vs absolute-player value once, then assert mirrored states and two-ply toy trees. Passing board-game `to_play=0/1` semantics into simultaneous movement without tests is a plausible silent inversion. |
| Simultaneous/joint-action row semantics | OpenSpiel treats simultaneous-node transitions as actions for all players together, and `legal_actions()` on a simultaneous node returns legal joint actions represented as flat integers; it also documents generic conversion of simultaneous games into turn-based games. See [OpenSpiel legal_actions](https://openspiel.readthedocs.io/en/stable/api_reference/state_legal_actions.html) and [OpenSpiel concepts](https://openspiel.readthedocs.io/en/latest/concepts.html#creating-sequential-games-from-simultaneous-games). | If our adapter stores fake half-steps, MuZero targets may correspond to adapter artifacts rather than one real CurvyTron physics tick. One replay row should mean one environment transition, or the bridge must be proven. |
| Self-play opponent freshness | AlphaZero says self-play games are generated from the latest continually updated network parameters, while OpenSpiel notes actors feed trajectories to a learner and warns slow/many actors can make data out of date relative to current weights. See [AlphaZero paper](https://ar5iv.labs.arxiv.org/html/1712.01815) and [OpenSpiel AlphaZero actors/learner](https://openspiel.readthedocs.io/en/latest/alpha_zero.html). | Log actor checkpoint, learner checkpoint, opponent source, replay insert time, and sampled age. "Self-play" should not hide fixed scripted, frozen, recent, latest, or live-same-policy opponents. |
| MCTS temperature and root exploration | AlphaZero/MuZero select real actions from root visit counts during self-play. MCTX exposes `temperature` for acting proportional to `visit_counts ** (1 / temperature)`, returns `action_weights` as the visit-prob target, masks invalid actions, and mixes root Dirichlet noise into priors. See [MuZero paper](https://ar5iv.labs.arxiv.org/html/1911.08265), [MCTX MuZero policy](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/policies.py#L1438-L1549), and [MCTX invalid-action masking](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/policies.py#L1970-L1989). | A no-learning run can be pure premature collapse: no root noise, near-zero collect temperature, wrong mask polarity, or all-invalid masks degenerating action selection. Record temp/noise/mask/action/visit counts per step. |
| Reward/value support scaling | MuZero Atari scales value/reward targets into categorical supports and inverts them at inference; Pohlen et al. motivate transformed Bellman targets for varying reward densities and scales. LightZero applies `scalar_transform`, `phi_transform`, and discrete support projection to target rewards and values. See the [MuZero paper](https://ar5iv.labs.arxiv.org/html/1911.08265), [Pohlen et al.](https://arxiv.org/abs/1805.11593), and [LightZero MuZero learn path](https://github.com/opendilab/LightZero/blob/main/lzero/policy/muzero.py#L405-L416). | Dump raw reward components, discounted return range, support bounds, and encode/decode examples. Saturated support or dense shaping can make survival, death, and win/loss indistinguishable to the value head. |
| Eval vs collect mode | LightZero's MuZero policy separates collect and eval paths: collect uses MCTS with root Dirichlet noise unless pure policy collection is enabled, while eval chooses the argmax action rather than sampling. See [LightZero collect path](https://github.com/opendilab/LightZero/blob/main/lzero/policy/muzero.py#L663-L780) and [LightZero eval path](https://github.com/opendilab/LightZero/blob/main/lzero/policy/muzero.py#L820-L898). | Keep stochastic collect, eval-MCTS, and raw-network curves separate. A checkpoint can collect useful diverse data while greedy/eval behavior still looks collapsed, or the reverse. |

## Pitfall Checklist

| Area | What to check in our code | Failure it can explain | Sources |
| --- | --- | --- | --- |
| Replay target construction | For every sampled position, verify targets are `(value, reward, policy)` for the same unroll index. MuZero's pseudocode builds value as discounted real rewards plus a bootstrapped future root value, emits the real reward as the reward target, and uses MCTS child visits as the policy target. Add a parity test against a tiny hand-authored trajectory. | Learner loss decreases or weights change, but value/reward heads learn shifted labels; policy imitates visits from the wrong timestep. | MuZero paper and official supplemental pseudocode links are on the arXiv page, including [`pseudocode.py`](https://arxiv.org/src/1911.08265v1/anc/pseudocode.py) ancillary file; the arXiv record lists the ancillary file set on the [MuZero page](https://arxiv.org/abs/1911.08265). |
| Initial vs recurrent reward alignment | Confirm the initial inference reward is ignored or padded according to the framework contract, while recurrent step `k` predicts the reward from applying action `k`. In LightZero-like code, inspect zero-padding/value-prefix conventions explicitly. | Off-by-one reward labels, especially if custom two-seat replay converts physical ticks into learner rows. | MuZero trains reward, value, and policy at each unrolled model step from stored trajectories ([MuZero arXiv](https://arxiv.org/abs/1911.08265)); implementation discussion and LightZero docs show scalar reward/value target transforms after target construction ([LightZero MuZero module](https://www.aidoczh.com/lightzero/_modules/lzero/policy/muzero.html)). |
| Replay row semantics for simultaneous games | Ensure one replay row equals one real CurvyTron physics tick. If the game needs both players' actions before the state changes, either use env-owned opponent/joint action, or prove a native replay bridge. Do not put fake half-steps into MuZero targets. | Training learns to predict artifacts of the adapter rather than game consequences; target rewards and policies no longer correspond to real decisions. | OpenSpiel distinguishes simultaneous legal joint actions from current-player sequential actions ([legal_actions docs](https://openspiel.readthedocs.io/en/stable/api_reference/state_legal_actions.html)). |
| Self-play opponent source | Log whether the opponent is live same checkpoint, latest checkpoint, recent frozen checkpoint, fixed scripted, random, or joint-action encoded. Treat each as a different experiment. | "Self-play" claims become ambiguous; a run may only prove learning against a fixed curriculum, not live two-seat self-play. | OpenSpiel AlphaZero actors generate self-play games and pass trajectories to the learner ([OpenSpiel AlphaZero](https://openspiel.readthedocs.io/en/latest/alpha_zero.html)); fictitious/self-play literature explicitly studies mixtures over historical play, not just the latest policy ([FSP, ICML 2015](https://proceedings.mlr.press/v37/heinrich15.html), [NFSP arXiv](https://arxiv.org/abs/1603.01121)). |
| Actor/learner staleness | Track actor checkpoint id, learner checkpoint id, episode duration, replay insert time, and sample age. If actors are slow, data can be generated by old weights for a long time. | Training chases obsolete policies; fresh checkpoints do not improve current collection because replay is dominated by stale data. | OpenSpiel warns that too many actors can make games finish slower and data more out-of-date relative to current weights, and uses fixed-size FIFO replay ([OpenSpiel AlphaZero](https://openspiel.readthedocs.io/en/latest/alpha_zero.html)). |
| Replay age/window | Plot sampled transition age and checkpoint id distribution, not just replay size. Confirm old games are evicted and that training ratio is not repeatedly overfitting a tiny early buffer. | Early bad play dominates; loss improves on replay but eval stays flat or regresses. | MuZero/AlphaZero are off-policy over self-generated replay; MuZero stores trajectories and samples positions for unrolled training ([MuZero arXiv](https://arxiv.org/abs/1911.08265)); OpenSpiel uses a FIFO replay buffer before updates ([OpenSpiel AlphaZero](https://openspiel.readthedocs.io/en/latest/alpha_zero.html)). |
| MCTS temperature and collection policy | Verify training collection samples from visit counts with nonzero temperature early enough, and eval uses the intended deterministic or low-temperature policy. Record temperature by move/step. | Premature collapse to one action; no exploration in sparse reward; misleading eval/collect mismatch. | AlphaZero samples from MCTS visit counts during self-play and uses exploration settings rather than raw greedy policy ([AlphaZero arXiv](https://arxiv.org/abs/1712.01815)). MCTX exposes `temperature` for acting proportional to `visit_counts ** (1 / temperature)` and returns visit-prob policy targets ([MCTX MuZero policy](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/policies.py)). |
| Root exploration noise | Confirm Dirichlet noise is applied only at the search root during training collection, not eval, and after legal-action masking semantics are correct. | Collection repeatedly follows policy priors too early; MCTS does not discover alternative actions. | AlphaZero-style search uses root exploration noise; MCTX's MuZero policy adds Dirichlet noise before search and then masks invalid actions ([MCTX MuZero policy](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/policies.py)). |
| Action masks and invalid actions | Assert mask length equals action space; valid/invalid polarity matches framework; all-invalid never happens except terminal; masked logits and sampled actions agree. Unit-test one state with each action invalid in turn. | Network/search selects illegal/no-op actions, or masks out the good actions. Since MCTX returns action 0 if all actions are invalid, this can look like a deterministic policy bug. | OpenSpiel legal actions are current-player or joint-action dependent ([legal_actions docs](https://openspiel.readthedocs.io/en/stable/api_reference/state_legal_actions.html)); MCTX documents invalid mask polarity and masks invalid actions in search/action selection ([MCTX policy](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/policies.py), [MCTX action selection](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/action_selection.py)). |
| `to_play` and player perspective | Decide if values are from ego/current-player perspective or absolute player-0 perspective, then enforce that everywhere: observations, rewards, value targets, MCTS backup, and eval. For non-board-game CurvyTron rows, do not pass board-game `to_play=0/1` semantics unless tested. | One side's good outcome becomes the other side's target; policy learns to help the opponent or oscillates. | AlphaZero uses a single network for both sides and board/current-player perspective is part of the algorithm contract ([AlphaZero arXiv](https://arxiv.org/abs/1712.01815)). Generic MCTX backs up scalar reward plus discounted child value; player sign/perspective must be supplied by the environment/model contract, not magically inferred ([MCTX search backup](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/search.py)). |
| Reward/value scale and support | Check raw reward range, return range, discount, support bounds, scalar transform, inverse transform, clipping, and loss type. Unit-test encode/decode for maximum plausible CurvyTron return. | Saturated support or clipped dense rewards make wins/deaths indistinguishable; value gradients vanish or explode. | MuZero predicts value and reward, and Atari-style variants use transformed categorical supports; Pohlen et al. target varying reward scales with a transformed Bellman operator ([Observe and Look Further arXiv](https://arxiv.org/abs/1805.11593)); LightZero applies `scalar_transform` and support projection to target rewards/values ([LightZero MuZero module](https://www.aidoczh.com/lightzero/_modules/lzero/policy/muzero.html)). |
| Sparse vs dense rewards | Run both a sparse terminal outcome probe and a dense-shaping probe with component logging. Ensure dense terms do not reward standing still, circling, suicide, or opponent mistakes in the wrong perspective. | Sparse-only reward may never bootstrap in early replay; dense reward may train the wrong proxy while eval outcome remains flat. | MuZero's value target is built from observed rewards plus bootstrapped value ([MuZero arXiv](https://arxiv.org/abs/1911.08265)); Pohlen et al. call out reward density/scale and long horizons as core Atari RL challenges ([Observe and Look Further](https://arxiv.org/abs/1805.11593)). |
| Greedy eval vs stochastic collection | Keep separate metrics for stochastic training collection, deterministic/low-temp eval, and greedy raw-network eval without MCTS. Do not use collection return as the only learning signal. | A stochastic collector improves replay diversity while greedy eval looks bad, or greedy eval hides that collection is still random. | AlphaZero/MuZero use MCTS visit distributions as improved policy targets rather than raw network actions ([AlphaZero arXiv](https://arxiv.org/abs/1712.01815), [MuZero arXiv](https://arxiv.org/abs/1911.08265)); MCTX returns both selected action and `action_weights` usable as policy targets ([MCTX README/policy docs](https://github.com/google-deepmind/mctx)). |
| Eval protocol | Pin seeds, opponent source, checkpoint ids, search sims, temperature, stochastic env settings, episode count, and confidence intervals. Evaluate against fixed/random/recent-frozen/live opponents separately. Save action histograms and survival/outcome curves. | Single-score or self-play-only eval can say "no learning" or "learning" depending on opponent drift and seed luck. | ALE protocol work emphasizes standardized evaluation, stochasticity, and reporting practices ([Machado et al., JAIR 2018](https://research.google/pubs/revisiting-the-arcade-learning-environment-evaluation-protocols-and-open-problems-for-general-agents/)); OpenSpiel evaluators play checkpoints against scalable MCTS opponents and log machine-readable metrics ([OpenSpiel AlphaZero](https://openspiel.readthedocs.io/en/latest/alpha_zero.html)). |
| Search/value backup contract | Inspect whether MCTS backup expects rewards/discounts in absolute environment coordinates or already perspective-adjusted coordinates. For two-player zero-sum CurvyTron, write a two-ply toy tree where the correct move is obvious and assert root visit preference. | MCTS can prefer losing lines even if the network is locally sensible, because child values are backed up with the wrong sign or discount. | MCTX backup is explicitly `reward + discount * leaf_value` into the parent, so perspective handling is external to this arithmetic ([MCTX search backup](https://github.com/google-deepmind/mctx/blob/main/mctx/_src/search.py)). |

## How this maps to our Pong mistake and CurvyTron mistake

This is intentionally narrow. The point is not "try more RL tricks"; it is to
separate artifacts we can verify locally from learning claims.

### Pong mistake

Local failure mode to avoid: mixing experiment identity and policy modes. A
stock/control Pong run, a survival-shaped Pong run, an eval-mode MCTS action,
and a collect-mode exploratory action are not interchangeable evidence.

Checklist:

| Check | Local evidence to inspect | Why it matters | Citation |
| --- | --- | --- | --- |
| Target construction | For stored rows, inspect the policy target. It should be the MCTS root visit distribution used for training, not merely the single action that collect mode executed. | If we audit only executed actions, we can wrongly call a checkpoint bad or good relative to the target actually used by MuZero. | AlphaZero/MuZero train policy toward search visit distributions ([AlphaZero](https://arxiv.org/abs/1712.01815), [MuZero](https://arxiv.org/abs/1911.08265)). |
| Eval mode vs collect mode | Record `mode`, MCTS simulation count, temperature/exploration settings, selected action, and visit counts in every Pong scorecard or target audit. | Eval-mode action quality and stochastic collect behavior answer different questions. | MCTX exposes both selected actions and action weights from search ([MCTX](https://github.com/google-deepmind/mctx)). |
| Reward scale/objective identity | Verify whether the run is stock sparse Atari Pong or a survival-shaped wrapper, and record reward schema in the artifact. | Survival-shaped reward can improve survival while not proving stock Pong learning. | MuZero value targets are built from observed rewards, so changing rewards changes the learned objective ([MuZero](https://arxiv.org/abs/1911.08265)). |
| Survival metric vs training reward | Compare reported survival curves with the actual reward used in replay. If survival is only an eval metric, do not present it as the optimized target. | A better survival curve can be useful telemetry without being evidence that the training reward was correct. | ALE protocol work stresses explicit evaluation settings and metrics ([Machado et al.](https://research.google/pubs/revisiting-the-arcade-learning-environment-evaluation-protocols-and-open-problems-for-general-agents/)). |

### CurvyTron mistake

Local failure mode to avoid: calling the custom two-seat path "self-play
MuZero" while it had taken over collection/replay/target construction from the
trusted stock path. The verifiable question is whether each replay row and
target still means what MuZero expects.

Checklist:

| Check | Local evidence to inspect | Why it matters | Citation |
| --- | --- | --- | --- |
| Target construction | Build one tiny hand-authored CurvyTron trajectory and assert reward, value, and policy targets for each sampled index. Include terminal and near-terminal rows. | The May failure can be explained by shifted or adapter-shaped targets even when gradients and losses look alive. | MuZero's target is discounted observed rewards plus a bootstrapped root value and MCTS policy target ([MuZero](https://arxiv.org/abs/1911.08265)). |
| Replay age | Add batch metadata for game id, insert step/time, actor checkpoint, learner checkpoint, and sampled position. Plot the sampled age distribution. | If replay is old or tiny, learning can chase stale early behavior. | OpenSpiel AlphaZero documents actor-generated games feeding fixed-size FIFO replay ([OpenSpiel](https://openspiel.readthedocs.io/en/latest/alpha_zero.html)). |
| Opponent staleness | Log opponent source per episode: fixed scripted, frozen checkpoint, recent checkpoint, live same-policy, or joint-action encoded. | "Self-play" is not one condition; opponent drift changes what a checkpoint result means. | Fictitious/self-play methods explicitly distinguish current play from mixtures over historical play ([FSP](https://proceedings.mlr.press/v37/heinrich15.html), [NFSP](https://arxiv.org/abs/1603.01121)). |
| Eval mode vs collect mode | For checkpoint curves, record whether actions came from eval MCTS, collect MCTS with exploration, or raw network logits. Keep survival/outcome/action histograms separate by mode. | A policy can collect diverse data while greedy/eval behavior remains collapsed, or the reverse. | AlphaZero-style training samples from search visits for self-play and evaluates under separate settings ([AlphaZero](https://arxiv.org/abs/1712.01815)). |
| Reward scale | Dump raw per-step reward components, episode return, discounted return range, support bounds, and encode/decode examples. | Dense survival/outcome reward may saturate support or reward a proxy that does not improve outcomes. | Transformed support methods are scale-sensitive by design ([Pohlen et al.](https://arxiv.org/abs/1805.11593)). |
| Action masks and `to_play` | Assert mask length/polarity, no all-invalid non-terminal masks, sampled action validity, and value perspective for mirrored player states. | A wrong mask or player perspective can make MCTS select legal-looking losing actions or train the ego to help the opponent. | OpenSpiel legal actions depend on current player/joint action context ([OpenSpiel legal actions](https://openspiel.readthedocs.io/en/stable/api_reference/state_legal_actions.html)); MCTX masks invalid actions before search selection ([MCTX](https://github.com/google-deepmind/mctx)). |
| Survival metric vs training reward | Report survival, outcome, and reward components separately. Confirm the checkpoint that improves survival also improves the reward actually stored in replay, or label it as metric-only. | CurvyTron can appear to improve by surviving longer against one opponent while not improving the intended game objective. | Evaluation protocols should specify metrics and conditions, not just a single score ([Machado et al.](https://research.google/pubs/revisiting-the-arcade-learning-environment-evaluation-protocols-and-open-problems-for-general-agents/)). |

## Fast Gates For CurvyTron

1. Target parity: one hand-authored trajectory proves reward/value/policy labels
   for terminal, near-terminal, and mid-game samples.
2. Mask parity: one state per action proves valid/invalid polarity and no
   all-invalid non-terminal rows.
3. Perspective parity: mirrored two-player states produce opposite/symmetric
   value targets as intended.
4. Replay freshness: sampled batch includes game age, actor checkpoint, and
   learner checkpoint histograms.
5. Eval split: every chart names trainer path, opponent source, temperature,
   search sims, seeds, and checkpoint id.
